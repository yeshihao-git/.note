## 1. 防抖 debounce
**逻辑**：多次触发只等停止后执行一次
场景：搜索输入、窗口resize、表单实时校验

## 2. 节流 throttle
**逻辑**：固定时间最多执行一次，匀速限流
场景：滚动监听、鼠标移动、拖拽、按钮重复点击

## 3. 延迟 setTimeout
**逻辑**：单纯延后执行一次，无合并限流
场景：弹窗延迟弹出、异步等待、DOM渲染后操作、简单延时提示

Base64 → Uint8Array 本质：把文本格式二进制还原成原始字节流

1. **文件上传/下载**
接口返回图片、PDF、Excel、音频base64，转字节数组再生成Blob预览、下载文件

2. **图片处理（Canvas/WebGL）**

4. **二进制通信**

5. **音视频处理**
音频/视频片段base64，转字节送入AudioContext、MediaRecorder

6. **二进制解析**
解析压缩包、自定义二进制协议、字体文件base64，转字节读取内部数据

### 3. 转换代码（浏览器可直接运行）
```js
// base64 转 Uint8Array
function base64ToUint8Array(base64) {
  const binary = atob(base64);
  const arr = new Uint8Array(binary.length);
  for (let i = 0; i < binary.length; i++) {
    arr[i] = binary.charCodeAt(i);
  }
  return arr;
}

const b64 = "SGVsbG8=";
const bytes = base64ToUint8Array(b64);
console.log(bytes);
```

### 4. 解码后 Uint8Array 结果
Uint8Array 就是 字节数组
```
Uint8Array(5) [72, 101, 108, 108, 111]
```

- 72 → H
- 101 → e
- 108 → l
- 108 → l
- 111 → o


# 一、先回答：为什么会有 Base64？
## 核心根源：二进制字节不能直接塞进纯文本载体
早期网络传输载体（HTTP 表单、URL、JSON、邮件、Cookie）**只安全支持 ASCII 可见字符**：
1. 原始二进制（图片、文件、加密字节）包含大量不可见字节：`0x00`、换行、0x80~0xFF 等
2. 这些特殊字节会被网关、邮件服务器、JSON 解析器截断、转义、丢弃、乱码
3. Base64 把任意二进制，统一转换成 **A-Z,a-z,0-9,+,/** 64 个安全可见字符**
好处：二进制能当普通字符串传输、存储，不会损坏。

举例：一张图片二进制不能直接放进 JSON，转 Base64 字符串就能存。

# 二、第二个问题：为啥不直接文本转 Uint8Array，非要走 Base64→Uint8Array？
分两种场景，完全不是一回事：
## 场景1：后端返回二进制文件/图片，用 Base64 封装在 JSON 里
后端限制：JSON 只能传字符串，**不能直接传原始二进制字节流**
- 方案A（可行）：后端二进制 → Base64 字符串（塞进JSON）→ 前端 Base64→Uint8Array
- 方案B（不可行）：后端二进制直接丢JSON，会乱码、丢失字节、解析报错

这种情况**没得选**，只能先Base64再转数组。

## 场景2：单纯普通明文（比如 "Hello"）
你完全可以**直接文本转 Uint8Array**，不需要 Base64：
```js
// 明文直接转字节（UTF-8），不需要Base64
new TextEncoder().encode("Hello") 
// Uint8Array(5) [72, 101, 108, 108, 111]
```
只有一种情况才需要明文先转Base64再转数组：
明文本身是**加密密文、二进制文件**，后端只能用字符串载体传输，才被迫用Base64中转。

# 三、一句话分清两种转换用途
1. **普通文字 → Uint8Array（TextEncoder）**
用途：把可读字符串转 UTF-8 字节，本地处理文本，无网络传输限制。
2. **二进制文件 → Base64字符串 → Uint8Array**
用途：二进制要通过JSON/URL/邮件等纯文本通道传输，必须Base64包装，前端再还原字节。

# 补充关键区别
- 直接文本转Uint8Array：编码 UTF-8，1字符≈1~3字节
- Base64中转：体积膨胀 1/3，只是二进制的“文本包装器”，不是字符编码。



## 一句话结论
底层网络**只传输二进制字节流**，JS 里用 `Uint8Array` 代表这一串字节；但上层接口有两种传输形式：二进制流 / Base64 字符串。

### 1. 真正标准高效的传输：二进制流（ArrayBuffer/Uint8Array）
- 请求头 `Content-Type: application/octet-stream`、Blob、WebSocket binary 模式
- 网络链路里跑的就是纯字节，前端拿到直接转 `Uint8Array`，**无需 Base64**
- 优点：体积无膨胀、速度快，文件上传下载、音视频、二进制通信首选

### 2. JSON 等文本协议的折中方案：Base64 字符串
JSON、URL、普通表单**不能直接携带原始二进制字节**，特殊字节会被转义、截断、乱码。
后端先把字节数组编码成 Base64 文本塞进 JSON，前端收到字符串再解码回 `Uint8Array`。
- 缺点：体积变大 1/3，多一次编解码开销

### 分层梳理
1. 物理层/传输层：永远只传 0/1 二进制字节，不存在字符串
2. HTTP 二进制接口：直接下发字节 → 前端 = Uint8Array
3. HTTP JSON 接口：二进制无法直接存放 → 包一层 Base64 字符串传输，前端还原字节数组

### 补充区分
- 传输原始文件、图片、加密数据：优先二进制流，不用 Base64
- 必须把二进制塞进 JSON 对象字段时，才被迫使用 Base64 中转


**计算属性** = 基于响应式依赖自动缓存的派生值。

- 依赖不变 → 读缓存，不重算
- 依赖变了 → 自动重算

## 剖面功能核心代码

```js
// 1. 必要导入
import vtkClipPolyData from '@kitware/vtk.js/Filters/Core/ClipPolyData';
import vtkPlane from '@kitware/vtk.js/Common/DataModel/Plane';
import vtkTriangleFilter from '@kitware/vtk.js/Filters/General/TriangleFilter';
import vtkMapper from '@kitware/vtk.js/Rendering/Core/Mapper';
import vtkActor from '@kitware/vtk.js/Rendering/Core/Actor';

// 2. 创建切面（定义平面位置和法线）
const plane = vtkPlane.newInstance({
  origin: [0, 0, 0],   // 平面经过的点
  normal: [1, 0, 0],   // 平面法线方向
});

// 3. 三角化（ClipPolyData 要求输入为三角形）
const triangleFilter = vtkTriangleFilter.newInstance();
triangleFilter.setInputConnection(yourInputPort); // 接你的模型数据源

// 4. 创建裁剪过滤器
const clipper = vtkClipPolyData.newInstance({
  clipFunction: plane,           // 用上面定义的平面裁剪
  generateClippedOutput: true,   // 同时输出两侧（保留侧 + 裁掉侧）
});
clipper.setInputConnection(triangleFilter.getOutputPort());

// 5. 保留侧（法线正方向）
const keptMapper = vtkMapper.newInstance();
keptMapper.setInputConnection(clipper.getOutputPort(0)); // port 0 = 保留侧
const keptActor = vtkActor.newInstance();
keptActor.setMapper(keptMapper);

// 6. 裁掉侧（法线反方向）
const clippedMapper = vtkMapper.newInstance();
clippedMapper.setInputConnection(clipper.getOutputPort(1)); // port 1 = 裁掉侧
const clippedActor = vtkActor.newInstance();
clippedActor.setMapper(clippedMapper);

// 7. 动态更新切面位置（核心交互函数）
function updateClipPlane(origin, normal) {
  plane.setOrigin(origin);
  if (normal) plane.setNormal(normal);
  renderWindow.render();
}
```

## 数据流

```
模型数据 → TriangleFilter → ClipPolyData → port 0 (保留侧) → Mapper → Actor
                                      → port 1 (裁掉侧) → Mapper → Actor
```

## 关键点

| 要素                            | 说明                                        |
| ----------------------------- | ----------------------------------------- |
| `vtkPlane`                    | 切面定义，`origin` 控制位置，`normal` 控制方向          |
| `generateClippedOutput: true` | 必须开启，否则只有保留侧没有裁掉侧                         |
| `getOutputPort(0)` / `(1)`    | 0=保留侧，1=裁掉侧                               |
| `vtkTriangleFilter`           | 裁剪前必须三角化，否则非三角面片会报错                       |
| `updateClipPlane`             | 交互时只需更新 plane 的 origin/normal，再 render 即可 |


## transform

CSS 属性，对元素做**2D/3D 变换**，包括：

| 值 | 效果 |
|---|---|
| `translate(x, y)` | 平移 |
| `rotate(45deg)` | 旋转 |
| `scale(1.5)` | 缩放 |
| `skew(10deg)` | 倾斜 |

可以组合：`transform: translate(10px, 20px) rotate(45deg)`

## translate

`transform` 的一个函数，**平移元素**。

```css
translate(100px, 50px)
/* 向右移100px，向下移50px */
```

## 关键特点

- **不影响文档流**：其他元素位置不变，不像 `margin` 会推开兄弟元素
- **性能好**：浏览器用 GPU 加速，比改 `top/left` 流畅
- **不改变布局**：元素原来的占位空间不变，只是视觉上移动了

所以拖拽场景都用 `transform: translate()` 而不是改 `top/left`。
